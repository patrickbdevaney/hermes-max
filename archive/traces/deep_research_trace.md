## 1. Core Implementation & Library

## 2. Data Structures & Serialization
- The `ProofInner` structure consists of `a` (G1), `b` (G2), and `c` (G1) elements [1].

- The `snarkjs2zkv` CLI utility converts proofs and verification keys generated by `snarkJS` to the zkVerify format [1].
- The `verify_proof()` function deserializes proofs and public inputs before verifying them against a verification key [1].

## 3. Curve Support & Encoding Details
- The `Curve` enum supports `Bn254` and `Bls12_381` [1].
- `G1` points are encoded as 64 bytes for BN254 and 96 bytes for BLS12-381 [1].
- `G2` points are encoded as 128 bytes for BN254 and 192 bytes for BLS12-381 [1].
- `Scalar` values are consistently encoded as 32 bytes [1].
- Elliptic curve points are represented in uncompressed form by concatenating the x and y affine coordinates [1].
- BN254 curve coordinates use little-endian encoding [1].
- BLS12-381 curve coordinates use big-endian encoding [1].
- Scalars are encoded in little-endian format [1].

## 4. Verification Process & Hashing
- The statement hash components include a context (`b"groth16"`), a verification key hash, and a public inputs hash [1].
- The `validate_vk` function checks field values and curve points [1].

## 5. Test Vectors & Reference Implementations
